# Interview Intelligence Platform

Веб-сервис для структурирования интервью: план вопросов (в т.ч. с AI), протоколы интервьюеров по ссылке, расхождения в оценках (правило **> 2 балла** считается в коде), AI-сводка и рекомендация, дашборд по вакансии, выгрузка CSV. Данные изолированы по HR-пользователю (email + пароль).

## Описание

HR создаёт вакансию и план интервью (вручную или с помощью AI), заводит «слоты» кандидатов и раздаёт интервьюерам персональные ссылки. Интервьюеры заполняют протокол без входа в систему. HR видит KPI, сравнение кандидатов, расхождения в баллах, AI-сводку и может выгрузить данные в CSV. Расширения вне MVP (ATS, почта, календарь) не входят в этот репозиторий — см. раздел Phase 3.

## База данных: где она находится и как работает

В проекте одна логическая схема данных (**PostgreSQL**), но **два типичных места хранения** — локально и в облаке. Подключение задаётся строкой **`DATABASE_URL`** (и при необходимости **`DIRECT_URL`** для миграций) в `.env` или в переменных окружения на хостинге. Сами пароли и хосты **не хранятся в Git** — только шаблон в `.env.example`.

### Локальная разработка

| Что | Как устроено |
|-----|----------------|
| **Где физически** | Контейнер **Docker** на вашем компьютере (файл [`docker-compose.yml`](./docker-compose.yml)). |
| **Образ** | `postgres:16-alpine`. |
| **Порт** | С хоста: **`localhost:5433`** → внутри контейнера порт `5432`. |
| **Имя БД / пользователь** | БД `testivor`, пользователь `testivor` (пароль задан в `docker-compose.yml`). |
| **Данные на диске** | Том Docker **`testivor_pg_data`** — файлы БД лежат в области данных Docker Desktop, а не в папке репозитория. |
| **Быстрый старт** | `npm run bootstrap` поднимает Postgres, подставляет `DATABASE_URL` в `.env`, прогоняет миграции и при желании сид. |

Локально приложение (**`npm run dev`**) через **Prisma Client** подключается к этому Postgres и выполняет запросы из server actions и API.

### Продакшен (работающий сайт)

| Что | Как устроено |
|-----|----------------|
| **Где** | **Облачный управляемый PostgreSQL** — в типичной настройке этого проекта используется **[Neon](https://neon.tech)** (отдельный проект в панели Neon, регион и имя инстанса видны только вам в консоли). |
| **Как приложение доходит до БД** | На **Vercel** в **Settings → Environment Variables** заданы `DATABASE_URL` (и при необходимости `DIRECT_URL`). Serverless-функции Next.js открывают соединения к этому хосту при каждом запросе (ограничения по пулу учитываются на стороне Neon / Prisma). |
| **Схема** | Та же, что локально: при каждом деплое выполняется **`prisma migrate deploy`** (см. [`scripts/vercel-build.mjs`](./scripts/vercel-build.mjs)) — на облачную БД накатываются файлы из **`prisma/migrations/`**. |

Итого: **локально** данные в Docker-томе на ПК; **на проде** — в выделенном облачном кластере Postgres (Neon или аналог), адрес и секрет только в переменных окружения.

### Как данные используются в коде

- **ORM:** [Prisma](https://www.prisma.io/) — схема в [`prisma/schema.prisma`](./prisma/schema.prisma), клиент в [`src/lib/prisma.ts`](./src/lib/prisma.ts).
- **Миграции:** SQL-файлы в `prisma/migrations/` — единый источник правды для структуры таблиц; локально создаёте новые через `npx prisma migrate dev`, на проде применяются автоматически при сборке.
- **Начальные данные:** [`prisma/seed.ts`](./prisma/seed.ts) — демо-пользователь, вакансии, кандидаты; запуск: `npm run db:seed` (на прод ту же команду выполняют вручную с тем же `DATABASE_URL`, если нужен демо-логин).

Связи в двух словах: пользователь (**User**) владеет вакансиями (**Vacancy**); у вакансии — план (**PlanBlock**, вопросы), кандидаты (**Candidate**), у кандидата — интервьюеры (**Interviewer**) с токеном ссылки; у интервьюера — протокол (**Protocol**) и оценки (**Score** и связанные сущности).

## Инструкция по приложению и сценарии тестирования

Ниже — как зайти без подготовки и что пройти руками. **Базовый URL продакшена:** [https://testivor.vercel.app](https://testivor.vercel.app). Локально замените на `http://localhost:3000`.

### Роли

| Роль | Вход | Задача |
|------|------|--------|
| **HR** | Email и пароль на `/login` | Вакансии, план, кандидаты, ссылки интервьюерам, AI, CSV |
| **Интервьюер** | Без аккаунта | Только по **персональной ссылке** `/interview/<токен>` — заполнить и отправить протокол |

### Доступ для проверки

1. Откройте сайт → **Войти** (`/login`).
2. **Демо-аккаунт** (если на базе выполняли `npm run db:seed` с тем же `DATABASE_URL`, что у стенда): **demo@demo.com** / **demo123**.
3. Если демо-логин не подходит — **Регистрация** (`/register`), затем работайте со своими данными (сценарии ниже всё равно применимы).

### Сценарий A — обзор демо-данных (≈5–10 минут)

Подходит, когда сид уже накатан на стенд.

1. Войдите как **demo@demo.com**.
2. На **дашборде** откройте вакансию **«Frontend-разработчик»** (основная демо-вакансия).
3. Оцените блок **KPI** и сводку по готовности протоколов.
4. Прокрутите **параметры вакансии** и **план интервью** (при необходимости разверните свёрнутые блоки).
5. В блоке **«Кандидаты»** откройте карточку, например **Борис Панов** — намеренно сильный разброс оценок между интервьюерами (для расхождений и AI).
6. На карточке кандидата проверьте: **ссылки интервьюеров**, кнопки **AI сводка**, **AI расхождения**, **сравнение** (если доступно), блок расхождений по правилу **> 2 балла**.
7. При желании откройте **каталог вакансий** (`/vacancies/browse`) — публичный просмотр объявлений (отдельно от кабинета HR).

### Сценарий B — путь интервьюера по прямой ссылке (без логина)

Используйте тот же базовый URL, что выше.

1. Откройте в **новом окне инкогнито** (чтобы не смешивать сессию с HR):
   - [https://testivor.vercel.app/interview/invite-demo-maria](https://testivor.vercel.app/interview/invite-demo-maria) — протокол для одного из интервьюеров по демо-кандидату **Алексей Иванов**.
2. Дополнительные демо-ссылки на ту же карточку (другие интервьюеры): `…/interview/invite-demo-oleg`, `…/interview/invite-demo-irina`.
3. Пройдите блоки плана: оценки и текстовые ответы; при **обязательных** блоках система не даст отправить незаполненное — ожидайте **центральное сообщение** на экране, а не только подсказку браузера.
4. Укажите **ФИО кандидата** в форме (если поле есть) — оно сохранится у HR в карточке.
5. Нажмите **Отправить протокол** — должно появиться подтверждение успешной отправки.

Если по этим токенам страница пустая или 404 — на стенде не выполняли **seed**; тогда используйте сценарий C и скопируйте ссылки с карточки кандидата после создания слота.

### Сценарий C — «чистый» HR: вакансия → слот → ссылка

1. Зарегистрируйтесь или войдите.
2. **Создать вакансию** (`/vacancies/new`) — заполните название и поля формы, сохраните.
3. На странице вакансии задайте **план интервью**: добавьте блоки/вопросы вручную или **Сгенерировать план (AI)** (если на сервере настроен ключ LLM — см. `.env.example` и [DEPLOY.md](./DEPLOY.md)).
4. Сохраните план кнопкой **«Сохранить план»**, если она есть в интерфейсе.
5. В блоке **«Кандидаты»** нажмите **«Добавить слот интервью»** (ФИО кандидата можно не вводить).
6. Откройте карточку нового кандидата → раздел **интервьюеры**: при необходимости переименуйте, добавьте до 4 человек, **Сохранить состав**.
7. Скопируйте **персональную ссылку** и откройте в инкогнито — заполните протокол по сценарию B.
8. Вернитесь в кабинет HR и убедитесь, что протокол отображается в карточке кандидата / KPI обновились.

### Сценарий D — выгрузка и ИИ

1. На странице вакансии нажмите ссылку **«Выгрузка CSV»** — должен скачаться файл (только для владельца вакансии, под сессией HR).
2. Кнопки **AI сводка**, **AI расхождения**, генерация плана: на **продакшене** они работают только если в Vercel заданы переменные с ключом и при необходимости `OPENAI_BASE_URL` / `OPENAI_MODEL`. Без ключа сервер вернёт понятную ошибку — это ожидаемо.

### Сценарий E — закрытие вакансии с наймом

1. На странице вакансии воспользуйтесь блоком **закрытия / найма** (выбор кандидата, подтверждение).
2. После закрытия с наймом интерфейс переходит в **режим только чтения** для редактирования плана и добавления слотов — проверьте, что кнопки недоступны, а просмотр данных сохраняется.

### Короткий чеклист для проверяющего

- [ ] Вход и выход, изоляция: второй пользователь не видит чужие вакансии.
- [ ] Создание вакансии и редактирование плана.
- [ ] Новый слот кандидата → ссылки → протокол без логина → успех после отправки.
- [ ] Обязательные блоки в протоколе нельзя «проскочить» без явного сообщения.
- [ ] Карточка кандидата: сводка, расхождения, сравнение (при ≥2 протоколах).
- [ ] CSV (если отображается).
- [ ] AI-функции при наличии ключа на стенде.

## Принятые решения

- **PostgreSQL вместо SQLite для продакшена:** на serverless (Vercel) файловая БД непригодна; одна и та же схема используется локально (Docker) и в облаке (Neon и т.п.).
- **Сборка на Vercel:** `npm run build` запускает `scripts/vercel-build.mjs` — сначала `prisma migrate deploy`, затем `next build`, чтобы схема на проде совпадала с миграциями.
- **Аутентификация:** NextAuth с Credentials и JWT; доступ к данным вакансий и кандидатов проверяется в server actions (`assertVacancyOwner` и аналоги).
- **Мутации и AI:** Server Actions в `src/actions/*`; вызовы LLM только с сервера, ключи — из переменных окружения (на Vercel — в панели проекта, не из репозитория).
- **Публичный протокол:** маршрут `app/interview/[token]/page.tsx`; валидация обязательных блоков на клиенте и сервере; у формы протокола `noValidate`, чтобы нативная HTML5-валидация не перехватывала submit до кастомной логики и модалок.
- **Расхождения:** детерминированное правило в `src/lib/discrepancies.ts` (порог **> 2** балла между интервьюерами).
- **ИИ:** провайдер OpenAI-совместимый; один ключ может задаваться под разными именами переменных (`OPENAI_API_KEY`, `OPENROUTER_API_KEY` и т.д.) — см. `.env.example`.

## Стек

- **Next.js 14** (App Router) + TypeScript + Tailwind
- **Prisma** + **PostgreSQL** (`DATABASE_URL`; локально — Docker, [Neon](https://neon.tech) и т.п.)
- **NextAuth** (Credentials, JWT)
- **OpenAI-compatible API**: ключ в `OPENAI_API_KEY` или `LLM_API_KEY` / `AI_API_KEY` / `OPENROUTER_API_KEY`; опционально `OPENAI_BASE_URL`, `OPENAI_MODEL` (дефолт — **gpt-4o-mini** или **openai/gpt-4o-mini** на OpenRouter). На **Vercel** те же переменные задаются в панели проекта (**Environment Variables → Production**), иначе ИИ работает только у вас локально, а не у гостей по ссылке.

## Быстрый старт

1. Установите [Node.js](https://nodejs.org/) LTS.
2. Зависимости: `npm install`.
3. **Самый простой локальный вариант** (нужен [Docker Desktop](https://www.docker.com/products/docker-desktop/)) — одна команда поднимает Postgres, создаст/починит `.env`, миграции и сид:

   ```bash
   npm run bootstrap
   ```

4. Запуск:

   ```bash
   npm run dev
   ```

   Без Docker: в корне файл **`.env`** с реальным `DATABASE_URL` (Neon и т.д.) и `NEXTAUTH_SECRET`. Можно скопировать `.env.example` → `.env` и заполнить вручную, затем `npx prisma migrate deploy` и `npm run db:seed`.

   После **изменений в `schema.prisma`**: `npx prisma migrate dev --name описание` (локально), затем закоммитьте папку `prisma/migrations`.

5. Откройте [http://localhost:3000](http://localhost:3000). Демо-аккаунт после сида: **demo@demo.com** / **demo123**. Пригласительные ссылки для протоколов: токены `invite-demo-maria`, `invite-demo-oleg`, `invite-demo-irina` (см. карточку кандидата или `prisma/seed.ts`).

## Архитектура

```
┌─────────────────────────────────────────────────────────────┐
│  Браузер HR (сессия NextAuth)                                │
│  app/(main)/* — дашборд, вакансии, кандидаты, план, AI, CSV  │
└───────────────────────────┬─────────────────────────────────┘
                            │ Server Components / Server Actions
┌───────────────────────────▼─────────────────────────────────┐
│  Next.js 14 App Router                                       │
│  • actions/* — мутации, права, AI                            │
│  • lib/prisma.ts — доступ к БД                              │
│  • lib/ai.ts, lib/discrepancies.ts — ИИ и расхождения        │
└───────────────────────────┬─────────────────────────────────┘
                            │ Prisma
┌───────────────────────────▼─────────────────────────────────┐
│  PostgreSQL (Neon / Docker / др.)                          │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  Браузер интервьюера (без логина)                            │
│  app/interview/[token] — протокол по секретному токену       │
└─────────────────────────────────────────────────────────────┘
```

**Каталоги:**

- `src/app/(main)/` — кабинет HR: дашборд, **создание вакансии** (`/vacancies/new`), страница вакансии (KPI, план, кандидаты), карточка кандидата (протоколы, AI, расхождения). У вакансии: приоритет, статус, срок закрытия, формат работы, единое поле «компетенции и пожелания к кандидату» (для рекрутера, карточки и ИИ), внутренняя заметка HR.
- `src/app/interview/[token]/` — публичная форма протокола для интервьюера.
- `src/actions/*` — server actions (мутации, AI).
- `src/lib/discrepancies.ts` — детерминированный расчёт расхождений.
- `src/lib/ai.ts` — вызовы LLM (JSON-ответы).
- `src/app/api/vacancies/[id]/export` — выгрузка CSV (сессия HR).
- `prisma/` — схема и миграции.
